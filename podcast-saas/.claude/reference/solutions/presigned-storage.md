# Design Doc тАФ Migrating podcast-saas to Presigned-URL Object Storage

> **Author:** fiji-advisor ┬╖ **Audience:** Product Owner + Architect ┬╖ **Status:** Proposal (read-only analysis; no source modified)
> **Reference model:** fiji (`/Users/admin/cebu/fiji`) тАФ a mature, horizontally-scaled platform whose storage design removes the entire class of problems podcast-saas hit with local-disk serving.

---

## 0. TL;DR for the Product Owner

podcast-saas currently stores and serves all user media (raw videos, HLS segments, thumbnails, captions, avatar b-roll, simulations) **through the Node app's own disk and HTTP routes**. That works, but it makes the app server the bottleneck and single point of failure for every byte a viewer watches, ties us to one machine, and only exists today because our Cloudflare R2 token is **read-only** so real uploads silently fall back to local disk.

fiji solved this once and never looked back: the **client uploads directly to the cloud** via a short-lived signed URL, the **viewer's browser/CDN pulls bytes directly from the cloud**, and the **app server only signs URLs and enforces per-object access** тАФ it never moves the bytes. This doc ports that model to our stack (Fastify + Postgres/Supabase + Next.js) as a **phased migration**, with the interim safety hardening **already shipped** and the full presigned model as the target.

The one true blocker is the **read-only R2 credential**. Until we have a *writable* object store, none of this lands. Recommendation: get real R2 write keys if available; otherwise adopt **Supabase Storage** (S3-compatible, HTTPS-only тАФ which fits the GoDaddy egress limits and the DB we already run there).

## 1. Where we are today (the problem, precisely)

### Current architecture: the app server is in the byte path for everything

| Concern | Today in podcast-saas | File evidence |
|---|---|---|
| Upload (raw video) | Browser тЖТ multipart POST to Node тЖТ **`uploadStreamWithFallback`** writes to local disk first, then *best-effort* re-uploads to R2 (fails silently, read-only token) | `controllers/v1/video.controller.ts:21,44,52`; `services/storage/uploadStreamWithFallback.ts` |
| Raw playback | Node `/video-proxy/*` streams from R2 (or redirects to `/video-raw/*` local) with Range support | `server.ts:299`, `R2StorageAdapter.streamObject` |
| HLS playback | `getPublicUrl()` returns a **Node proxy URL** `/hls-proxy/{key}`; Node fetches each segment from R2 (or redirects to `/hls-public/`) | `server.ts:171`; `R2StorageAdapter.getPublicUrl():147` |
| Thumbnails / images / captions / avatar-circles | Served from `/local-storage/*` (public prefixes, no auth) | `server.ts:127` |
| Simulations | `/sim-public/*` (local) or R2 public URL | `server.ts:342` |
| Delete | `deleteWithFallback` / `deleteWithPrefixFallback` (R2 delete is a **no-op** under read-only token; local bytes are the real ones) | `video.controller.ts:194,198` |

**Net effect:** every viewer's video stream, every HLS segment, every thumbnail flows **through our one Node process**. That process buffers/relays the bytes, competes with ffmpeg transcodes for CPU and sockets, and cannot be replicated because the canonical media lives on *its* local disk.

### Why fiji doesn't have any of this

fiji's `StorageProxyHandler.ts` and `StorageController.ts` make three structural choices that podcast-saas only partially has:

1. **Uploads never touch the app's disk.** `POST /storage/upload-url` (`StorageController.ts:44`, `@Security('firebase')`) returns a **presigned PUT URL** with a *server-constructed* key; the browser PUTs straight to S3/GCS/Azure. No bytes through Node тЖТ no buffering, no disk, no memory blowup.
2. **Downloads are signed URLs or CDN-public**, generated by `StorageService.getPresignedUrl()` (`StorageService.ts:415`, 1-hour expiry, 30-min in-memory cache `presignedUrlCache:31`). The browser fetches from cloud/CDN.
3. **When bytes must pass through Node** (e.g. relative-URL artifacts, headless capture), the `/storage/proxy/{key}` handler treats the wildcard as a **cloud object key**, not a filesystem path тАФ so OS path traversal is *structurally impossible* тАФ and runs **per-object authorization** (`checkArtifactAccess` / `checkVideoAccess`, `StorageProxyHandler.ts:234,326`) gating on `artifact.isPublic` OR owner OR admin OR a **scoped token** OR localhost.

podcast-saas already has the *abstraction* (`StorageService` interface with `getPresignedUploadUrl` / `getPresignedDownloadUrl`) and already ported the safety hardening (see ┬з10). What it lacks is (a) a **writable** store and (b) actually **handing the signed URLs to the browser** instead of proxying every byte.

## 2. Uploads тАФ what changes for the editor

### Target flow (fiji-style, three steps)

```
1. Editor calls  POST /api/v1/projects/:id/videos/upload-url   { filename, contentType, sizeBytes }
   тЖТ backend authorizes (owner of :id), builds the key  videos/{projectId}/{uuid}.{ext},
     inserts a video_files row (status: 'uploading'), returns { uploadUrl, storage_key, videoFileId }
2. Browser does  PUT {uploadUrl}  with the raw file body тАФ straight to R2/Supabase, NOT through Node.
3. Browser calls POST /api/v1/projects/:id/videos/:videoFileId/confirm
   тЖТ backend flips status тЖТ 'uploaded', enqueues the HLS transcode (write-path enqueue, not read-path).
```

This is exactly fiji's `getUploadUrl` shape (`StorageController.ts:44-90`) plus an explicit confirm step (fiji folds confirmation into artifact creation; we keep it separate because our transcode pipeline is a distinct stage).

### Product-visible impact

- **Faster, more reliable uploads of large files.** Today a 2 GB lecture upload is streamed *into our Node process* (10 GB multipart limit, `server.ts:103`), held on our disk, then re-uploaded. With presigned PUT the file goes **once**, directly to cloud, at the cloud's ingest bandwidth тАФ not throttled by our single instance.
- **The app server stops being a bottleneck.** Multiple creators uploading simultaneously no longer contend for our process's memory/sockets.
- **The local-disk fallback fragility disappears.** No more "upload succeeded but the bytes only exist on one ephemeral container disk" (the current `uploadStreamWithFallback` reality under a read-only token). The cloud is the single source of truth.
- **Editor UX:** a real progress bar driven by the browser's own PUT (`xhr.upload.onprogress`), resumable/multipart for very large files (S3/R2 multipart тАФ a later enhancement), and clean failure semantics (a failed PUT is retried by the browser, not silently swallowed by the server as it is today, `uploadStreamWithFallback.ts:46`).

### Large-file handling

- Single-PUT presigned URLs are fine to a few GB. For the 10 GB ceiling we currently advertise, the target is **S3/R2 multipart upload** (presign each part, browser uploads parts in parallel, backend completes the multipart). Defer to a later phase; single-PUT covers the common case (lectures/podcasts) first.
- **Egress note:** the GoDaddy host only allows outbound 80/443. Presigned PUT/GET are plain HTTPS (443) тЖТ fits. The browser-to-cloud PUT doesn't traverse our container at all, so the egress limit is irrelevant to upload throughput.

## 3. Playback тАФ HLS/video via signed GET / CDN vs the Node proxy

### What changes

Today `getPublicUrl()` hands the player a **proxy URL on our own backend** (`/hls-proxy/{key}`, `R2StorageAdapter.ts:147`), so **our Node process fetches and relays every .ts segment** for every viewer. The HLS proxy was even a P1 heap-pressure finding before it was switched to streaming (`server.ts:195` now pipes instead of buffering).

Target: `getPublicUrl()` returns either
- a **CDN-public URL** for HLS segments/manifests (segments are immutable, public, cacheable тАФ ideal for a CDN), **or**
- a **short-lived presigned GET URL** for private raw video (mirrors fiji `getPresignedUrl`, `StorageService.ts:415`).

Either way, **the viewer's player pulls bytes from the object store / CDN, not from our app**.

### Buffering / latency / reliability implications

- **Latency & throughput:** HLS from a CDN edge beats a single origin Node process on every axis тАФ lower time-to-first-segment, parallel segment fetch, no contention with transcodes.
- **Reliability:** today an HLS stream dies if our one instance is busy, restarting, or OOMs. CDN/object-store delivery is independent of app health тАФ a redeploy no longer interrupts in-flight playback.
- **Buffering:** the player buffers ahead by fetching future segments directly; we remove the double-hop (browserтЖТNodeтЖТR2) and its abort/refetch leakage class (`fiji-storage-006`).
- **The CORS workaround goes away.** The proxy exists today because `pub-*.r2.dev` ignores `PutBucketCorsCommand` (`server.ts:169` comment). With a properly CORS-configured bucket/CDN (or Supabase Storage, which honors CORS config), segments load cross-origin directly and `/hls-proxy` can be retired.

### What stays a proxy (deliberately)

Captions are **stored in Postgres** (`video_files.captions_vtt`) and served from the DB (`player.controller.ts:/videos/:videoId/captions.vtt`), not object storage тАФ keep that; it's simpler and already access-gated. Only *media bytes* move to direct delivery.

## 4. Anonymous viewers тАФ the "Ask" avatar + public course pages

This is the subtlety that a naive "just use presigned URLs" plan breaks, and it's exactly what fiji's **scoped-token + optional-auth proxy** model is for.

### Who is anonymous here

- The **avatar "Ask" feature** serves viewers who are **not logged in** тАФ `POST /api/v1/avatar/start`, `/avatar/visual/analyze`, `/avatar/image/analyze`, `/avatar/memory` are all **unauthenticated today** and act on a client-supplied `projectId` / `sessionKey` (`avatar.controller.ts:57,93,112,156`).
- **Public course pages** (`/c/...`) and free-project viewers hit **optional-auth** endpoints тАФ `player-config` and `captions` use `firebaseAuthOptionalMiddleware` (`player.controller.ts:16,45`), so `request.dbUser` may be `null`.

### How presigned URLs still serve them (fiji's answer)

fiji serves anonymous viewers of **public** content without any login because access is decided by the **object's own `isPublic` flag**, not by who is asking (`StorageProxyHandler.ts:265` "Allow access if artifact is public"). Three mechanisms, in order of preference for us:

1. **Public content тЖТ CDN-public or presigned GET handed to anyone.** For a *published / public* project, the player-config endpoint returns media URLs that work for an anonymous browser. No auth needed because the *content* is public. This is the common case for course pages and the "Ask" viewer.
2. **Private/draft content viewed by a non-owner via a share link тЖТ scoped token.** fiji's `ArtifactTokenService` (`ArtifactTokenService.ts:47`) issues a **1-hour RS256 JWT scoped to one artifactId**, passed to the sandboxed viewer and accepted by the proxy (`StorageProxyHandler.ts:207 hasValidArtifactToken`). Port: when a viewer opens a `?share=<token>` link, the backend mints a **project-scoped capability token**; the optional-auth proxy (and any presigned-URL minting) accepts that token in lieu of a logged-in session. This replaces our current `share_token` (a static unguessable string on `projects.share_token`) with a short-lived, signed equivalent for *media* access while keeping the human-shareable link.
3. **Avatar capability tokens (fixes a current IDOR).** The avatar's `sessionKey` is currently client-supplied and unauthenticated (`avatar.controller.ts:156`; frontend derives it from `Math.random()`), which is a cross-tenant IDOR + cost-DoS (review `security-004/005`). Port fiji's scoped-token idea: the backend issues a **server-signed, project-scoped session capability** when the viewer opens a project, and the avatar endpoints validate it. The viewer still never logs in тАФ the token *is* the anonymous identity, but it's unforgeable and scoped.

**Key product guarantee:** the "Ask" flow and public course pages keep working for logged-out users. We are not adding a login wall; we are replacing *guessable/forgeable* identifiers with *server-issued scoped tokens*, and serving public media as genuinely public URLs.

## 5. Privacy / access control тАФ public vs private media

### The model fiji uses

Access is a **property of the record, evaluated at serve time**, not a path prefix. `checkArtifactAccess` (`StorageProxyHandler.ts:234`) and `checkVideoAccess` (`:326`) resolve the artifact from the DB and allow iff **`isPublic`** OR **owner** OR **admin** OR **valid scoped token** OR **localhost** (internal services). Anything else тЖТ **403**.

Contrast podcast-saas today: public media is decided by **path prefix** тАФ `/local-storage/*` treats `playlist-banners/`, `thumbnails/`, `crop/`, `images/`, `audio/`, `captions/`, `avatar-circles/` as public (`server.ts:127`), and HLS is "public if the key starts with `hls/`". That's coarse: it can't express "this *specific* video is private."

### The open product decision this forces (call-out)

> **PRODUCT DECISION REQUIRED:** Should a free/draft project be **world-readable by id** (anyone with the UUID can watch), or **gated** by `published` / `share-token` / an explicit `is_public` flag?

This is unresolved in the code today and it's the crux of the whole privacy story:

- `player.controller.ts` gates **only** on `access_type === 'paid'` (`:24`). A **free draft** project is therefore **fully readable by anyone who knows/guesses its id** тАФ there is *no* published/visibility check (review `fiji-contracts-002`).
- The schema has `projects.status` (a **pipeline** enum: draftтЖТready, `schema.ts:23`), `projects.share_token` (unique, `:167`), and `projects.access_type` (free/paid, `:170`) тАФ but **no `is_public` column**. `courses` has the real `published`/`published_at` publish state (`schema.ts:91,650`).

**Recommendation (architect's view):** introduce an explicit **per-project visibility** the way fiji has `isPublic` тАФ e.g. an `is_public BOOLEAN` (or a `visibility` enum `private | unlisted | public`) on `projects`, decoupled from the pipeline `status`. Then:
- **public** тЖТ media URLs served to anyone (anonymous OK).
- **unlisted** тЖТ served only with a valid `share_token` / scoped token.
- **private/draft** тЖТ owner (or scoped token) only.

This single column is what lets the storage layer answer "may this requester see this object?" the way fiji does тАФ and it must be decided **before** we hand out presigned URLs, because a presigned GET URL, once minted, is bearer-authority (see ┬з6). The migration is small (one nullable column + backfill `is_public = (status='ready' AND <listed>)`), but the **policy** is a product call, not an engineering one.

### How enforcement works under presigned URLs

- **Public objects** тЖТ a CDN-public URL (no signing) or a long-ish presigned GET. Cheap, cacheable.
- **Private objects** тЖТ backend mints a **short-lived** presigned GET **only after** an access check (owner / scoped token), exactly when building player-config. The check lives in **our code** (a `requireProjectAccess(projectId, requester)` helper, the Fastify analogue of fiji's `checkVideoAccess`), and Drizzle replaces fiji's Mongoose `Artifact.findById`.

## 6. URLs & sharing тАФ what media links look like

| Aspect | Public objects (published media) | Private objects (drafts, paid, owner-only) |
|---|---|---|
| URL form | CDN-public `https://cdn.тАж/hls/{id}/master.m3u8` (stable) | Presigned GET, **expiring** (e.g. 1 h), `тАж?X-Amz-Signature=тАж` |
| Share link | The **app** share URL (`/c/{slug}` or `?share={token}`) is the durable thing; it resolves to fresh media URLs server-side on each load | Same: the share/scoped **token** is durable-ish (or refreshable); the underlying presigned media URL is short-lived and minted per page load |
| Hotlinking | Possible for public CDN assets (that's the point тАФ they're public). Mitigate abuse with CDN referrer rules / cache rules if needed | **Resistant** тАФ a presigned URL expires; a leaked link dies in ~1 h |
| SEO / Open-Graph thumbnails | **Use a stable CDN-public URL** for `og:image`. Crawlers (Slack/Twitter/Google) won't follow expiring signed URLs and can't authenticate тАФ thumbnails must be public + stable | n/a (private content shouldn't be in OG tags) |
| Link longevity | Stable | Intentionally ephemeral; the app re-mints on demand |

**Design rule (matches fiji):** the **human-shareable URL is always an *app* URL**, never a raw signed media URL. The app resolves it to fresh media URLs on each request. This keeps share links permanent while letting the underlying media URLs expire тАФ and it's why thumbnails for OG/SEO must be a **separate, deliberately-public, stable** object (our `thumbnails/` prefix is already treated as public, `server.ts:127`; keep those CDN-public and **not** presigned, or social cards and Google break).

> **PRODUCT DECISION:** confirm that **thumbnails and OG images are always public + stable URLs**, even for projects that are otherwise private/unlisted. (Standard practice: the poster frame is public so the card renders; the video itself stays gated.)

## 7. Costs тАФ where the money moves

| Cost driver | Today (Node-served) | Presigned / CDN model |
|---|---|---|
| **Egress bandwidth** | Every viewed byte leaves **our app server** twice for HLS (R2тЖТNodeтЖТviewer). On GoDaddy that's our container's 443 egress and CPU | Bytes leave the **object store / CDN** directly. R2 has **zero egress fees**; Supabase/S3 charge egress but a CDN collapses it via cache hits |
| **App compute** | Node spends CPU/sockets relaying media *and* signing | Node only **signs URLs** (cheap) + serves JSON config. Far smaller instances suffice |
| **Signing overhead** | n/a | A signature is an HMAC тАФ microseconds. fiji **caches** presigned URLs 30 min (`StorageService.ts:31`) to avoid re-signing hot objects; we'd do the same |
| **Storage at rest** | Local disk on the container (ephemeral!) + partial R2 | One durable bucket. Predictable per-GB |
| **CDN** | none (we *are* the origin) | Cache hits make repeat views ~free; first byte from origin once |

**Cost-shape summary:** we **move bandwidth off the app server onto the object store/CDN**, shrink the app tier, and (with R2's no-egress pricing or CDN caching) likely **lower** total media-delivery cost while removing the scaling cliff. The new line items are storage-at-rest and (for S3/Supabase) egress тАФ both far cheaper than scaling Node instances to relay video. Signing is negligible, especially cached.

## 8. Future scalability тАФ what this unlocks (and what else must change)

The presigned model is the **precondition for running multiple stateless app instances** тАФ fiji's whole horizontal story (`fiji.md`: "presigned direct-to-cloud тЖТ app servers stay stateless and bandwidth-light"). But storage is necessary, **not sufficient**. To actually run N instances, these per-process assumptions must also go (all flagged in the latest review):

1. **Local-disk canonical storage тЖТ object store.** (This doc.) As long as canonical media is on one container's disk, instance #2 can't serve it.
2. **In-process state must move out.** `getStorageAdapter` is a module-level singleton (`getStorageAdapter.ts:6`); the caption/crop pipelines use in-process `_inFlight` Sets and fire-and-forget transcodes (review `arch-008`). For N instances: dedup via a **Postgres advisory lock** (hot caption path) or a real job queue, not a per-process Set.
3. **ffmpeg must be bounded and ideally externalized.** Transcodes are fire-and-forget with **no global concurrency cap** (review `fiji-storage-002`); fiji uses a single poll-loop dispatcher with a pool as the sole limiter (`JobDispatcher`). On a multi-instance world this becomes a shared queue with a worker tier (the long-term home for HLS, captions, crop, thumbnails).
4. **Graceful shutdown** (already added, `server.ts:recoverStuckTranscodes`) plus stale-row recovery so a redeploy of any instance doesn't strand `hls_status='processing'`.

**Sequencing:** presigned storage **first** (removes the disk anchor), then the queue/worker extraction, then flip on multiple instances. Trying to scale out before storage moves is impossible; doing storage first makes the rest incremental.

## 9. The R2-read-only blocker тАФ the one thing that gates everything

**Nothing in this doc lands while the R2 token is read-only** (`PutObject` тЖТ `AccessDenied`), because presigned **PUT** URLs require write authority and presigned **GET** requires the object to actually be in the bucket. Today every "upload to R2" is a silent no-op that falls back to local disk (`uploadStreamWithFallback.ts:46`), and every R2 "delete" is a no-op (review `backend-003`). Two ways out:

### Option A тАФ real R2 write credentials (preferred if obtainable)
- **Pros:** keeps the `R2StorageAdapter` we already wrote (`R2StorageAdapter.ts`); **zero egress fees** (best media-bandwidth economics); S3-compatible so the AWS SDK and presigned-URL code already work; HTTPS-only (fits GoDaddy 80/443).
- **Cons:** must obtain a Cloudflare token with `Object Read & Write` on the bucket; `pub-*.r2.dev` CORS quirk means HLS either keeps a thin proxy or fronts R2 with a CDN/custom domain that honors CORS.
- **Effort to unblock:** **S** (rotate the token, set `R2_*` to real values; `getStorageAdapter` already detects real vs placeholder creds, `getStorageAdapter.ts:13`).

### Option B тАФ Supabase Storage (best fit if R2 write keys aren't available)
- **Why it fits us specifically:** the app **already runs on Supabase/Postgres**, and Supabase Storage is **S3-compatible and strictly HTTPS** тАФ which directly answers the GoDaddy egress limit (80/443 only) *and* the open review concern that Supabase's pooler port might be blocked (Storage is over 443, not a DB port). It honors CORS config, so the `/hls-proxy` workaround can retire cleanly.
- **Pros:** one vendor for DB + storage + auth-adjacent tooling; native RLS-style access policies; HTTPS-only; S3-compatible endpoint works with our existing `StorageService`/`getSignedUrl` code (add a `SupabaseStorageAdapter` behind the same interface).
- **Cons:** has egress fees (mitigate with a CDN); a new adapter to write and test (small тАФ the interface exists); ecosystem less "media-CDN-native" than R2+Cloudflare.
- **Effort:** **M** (new adapter implementing the 10-method `StorageService` interface + config + tests).

### Provider call
**Get real R2 write keys if the account allows it (Option A) тАФ least new code, best egress economics. If write keys are genuinely unavailable, go Supabase Storage (Option B)** because it is HTTPS-only (egress-limit-friendly), co-located with our existing Supabase data plane, and S3-compatible so it drops into the same abstraction. Either way, **keep the `StorageService` interface** so the provider stays swappable (don't hard-code a vendor; do *not* replicate fiji's full multi-cloud тАФ one writable store is enough).

## 10. Phased migration plan

**Phase 0 тАФ Interim hardening тАФ тЬЕ ALREADY DONE** (verified in current source)
- `safeLocalPath()` + `keyHasTraversal()` containment on **every** serve route, killing the P0 traversal class (`server.ts:138,158,175,303,349`; `services/storage/pathSafety.ts`).
- Auth on private local prefixes; `/local-storage/upload/*` gated behind `NODE_ENV!=='production'` **and** auth (`server.ts:366,369`).
- Range-streaming for raw video, `nosniff` + octet-stream default content-type (`server.ts:62,213`).
- HLS proxy **pipes** instead of buffering; aborts upstream on client disconnect (`server.ts:183,195`).
- `deleteWithFallback` / `deleteWithPrefixFallback` for GC; graceful **stuck-transcode recovery** on startup (`server.ts:67`).
- **Product impact:** none visible. **Risk:** low. **Effort:** done.

**Phase 1 тАФ Unblock writes (the gate)** тАФ *Effort: **S** (R2) / **M** (Supabase)*
- Resolve ┬з9: real R2 write creds **or** add `SupabaseStorageAdapter` behind `StorageService`.
- Verify a real round-trip: presigned PUT тЖТ object in bucket тЖТ presigned GET works.
- **Product impact:** uploads stop silently living only on local disk тЖТ durability win, no UI change. **Risk:** low-med (credential/CORS config). **Decision needed:** provider (R2 vs Supabase).

**Phase 2 тАФ Presigned uploads (direct-to-cloud)** тАФ *Effort: **M***
- Add `POST тАж/videos/upload-url` (owner-checked, server-built key) + `тАж/confirm`; switch the editor to **PUT direct to cloud** with a real progress bar. Retire the multipart-through-Node path for video (keep small images as-is initially).
- Bucket CORS for PUT locked to app origins (we already do this, `R2StorageAdapter.ensureBucketCors:201`).
- **Product impact (visible):** faster, more reliable large-file uploads; honest progress + failure. **Risk:** med (editor flow rewrite). Remove the `uploadStreamWithFallback` fragility for video.

**Phase 3 тАФ Direct playback (signed GET / CDN)** тАФ *Effort: **M***
- `getPublicUrl()` returns CDN-public (public objects) or presigned GET (private); front HLS with a CORS-honoring CDN/custom domain and **retire `/hls-proxy`**. Add fiji-style presigned-URL caching (`StorageService.ts:31`).
- **Product impact (visible):** smoother playback, less buffering, playback survives app redeploys. **Risk:** med (CDN/CORS setup, cache-busting on re-transcode тЖТ version the HLS key per run, review `fiji-storage-008`).

**Phase 4 тАФ Access control + visibility model** тАФ *Effort: **M***
- Add per-project visibility (`is_public` / `visibility` enum) + `requireProjectAccess()` (the Fastify analogue of `checkVideoAccess`). Mint **scoped capability tokens** for share links and the avatar session (replacing `Math.random()` sessionKey тАФ fixes review `security-004/005`). Gate player-config/media on published-or-shared, not just `access_type==='paid'` (fixes `fiji-contracts-002`).
- **Product impact (visible):** drafts stop being world-readable by id; share links and "Ask" keep working for anonymous viewers via tokens. **Risk:** med тАФ **needs the ┬з5/┬з6 product decisions first.** **Decision needed:** the draft-visibility policy and the thumbnail-always-public rule.

**Phase 5 тАФ Statelessness + worker tier (scale-out)** тАФ *Effort: **L***
- Externalize ffmpeg/caption/crop into a bounded queue + worker; replace in-process `_inFlight` dedup with a pg advisory lock or queue; then enable multiple app instances.
- **Product impact:** capacity & resilience (invisible to users until load grows). **Risk:** high (concurrency, job semantics). This is the big one; everything above makes it *possible*.

## 11. Risks & trade-offs (cross-cutting)

- **Don't over-build to fiji parity.** fiji is multi-cloud (S3+GCP+Azure); we need **one writable store**. Port the *abstraction*, not the breadth (already true тАФ our `StorageService` is single-provider-swappable).
- **Presigned GET is bearer authority.** Once minted, anyone with the URL can fetch until expiry. Keep TTLs short for private content; never put private media in OG tags; the share *token* (not the signed URL) is the durable artifact.
- **Cache-busting on re-transcode.** HLS keys are reused in place today (`hls/{video_file_id}`); a CDN will cache stale segments. Version the key per transcode run and flip a pointer (review `fiji-storage-008`).
- **Migration is dual-read during transition.** Existing media is on local disk; new media in cloud. Keep the local serve routes (now hardened) until a backfill copies legacy objects to the bucket, then retire them.
- **GoDaddy egress (80/443) constrains provider choice**, not the design тАФ both R2 and Supabase Storage are HTTPS-only and fine; an arbitrary-port store would not be.

## 12. Verification (how we'll know each phase is good)

- **Phase 1:** integration test тАФ presigned PUT then presigned GET returns identical bytes against the real provider; `getStorageAdapter()` resolves the cloud adapter (not local) with real creds.
- **Phase 2:** editor uploads a >1 GB file with **no bytes through Node** (assert app memory flat during upload); `video_files` row transitions uploadingтЖТuploadedтЖТtranscode.

## 13. Recommendation (one paragraph)

Adopt fiji's presigned-URL object-storage model, phased: **first unblock writes** (the only true blocker) by obtaining **real R2 write credentials** тАФ least new code, zero egress fees, and our `R2StorageAdapter` is already written тАФ or, if those keys are genuinely unavailable, by adding a **Supabase Storage** adapter behind the existing `StorageService` interface (S3-compatible, HTTPS-only, co-located with the Supabase data plane we already run, and a clean fit for GoDaddy's 80/443 egress limit). Then move **uploads to direct-to-cloud presigned PUT** (durability + large-file UX), **playback to CDN/presigned GET** (retire the per-byte Node proxy, survive redeploys), and **introduce an explicit per-project visibility flag plus scoped capability tokens** so anonymous "Ask" and public-course viewers keep working while drafts stop being world-readable тАФ which requires a product decision on draft visibility and on keeping thumbnails public for SEO/OG. The interim path-traversal/auth hardening is already shipped, so this is a forward migration, not a rescue; statelessness/worker-tier work comes last and is *unlocked* by the storage move, not blocked on it.
